# SISFIT Coursepack Skill

一个面向身体教育老师的中文 Skill，用于把身体主题整理成边界清晰、可交付、可销售的主题课包骨架。

English version: [README_EN.md](./README_EN.md)
Contributing guide: [CONTRIBUTING.md](./CONTRIBUTING.md)

## 它解决什么问题

很多老师已经有主题、动作素材和用户痛点，但缺少一套稳定的输出框架，把这些内容整理成：

- 主题定位
- 适合 / 不适合人群
- 课程目标
- 课程结构与流程
- 课堂口令方向
- 表达边界清单
- 预热内容方向
- 销售页结构

这个 Skill 的目标不是给医学判断，而是帮助老师完成健康教育内容策划和课包产品化表达。

## 适用场景

- 线下主题课设计
- 线上课或直播课结构整理
- 小红书 / 朋友圈 / 公众号预热内容准备
- 课包销售页骨架生成
- 健康教育类课程的风险表达降火

## 不适用场景

- 医学诊断
- 疾病判断
- 疼痛原因定论
- 治疗建议
- 康复处方
- 个体化医疗建议

如果输入涉及外伤、明显肿胀、麻木、发热、夜间痛、活动明显受限等情况，Skill 应先提示线下就医或咨询合格专业人员。

## 安装

将本目录放入本地技能目录：

```text
~/.trae/skills/sisfit-coursepack/
```

目录中至少应包含：

- `SKILL.md`
- `README.md`
- `README_EN.md`
- `CONTRIBUTING.md`
- `examples.json`
- `boundary_checklist.md`
- `xiaohongshu_post.md`

## 调用方式

当你要做以下任务时调用这个 Skill：

- 生成一个身体主题课包骨架
- 把身体主题整理成可卖的课程结构
- 输出主题课流程、口令和表达边界
- 生成这个主题的小红书预热方向或销售页骨架
- 只输出预热内容、销售页骨架或课程流程，而不是完整课包

可直接使用类似表达：

- “帮我做一个身体主题课包骨架”
- “把这个主题整理成 60 分钟线下课结构”
- “帮我输出这个主题的课堂口令和销售页结构”

## 输入模板

```text
老师类型：
目标人群：
身体主题：
课程时长：
使用场景：
我想传递的核心观点：
学员最常问的问题：
已有动作或练习素材：
是否涉及特殊人群：
```

## 输出模式

Skill 支持 5 种输出模式：

1. `full_package`：完整主题课包骨架
2. `class_flow_only`：只输出课程结构、流程、口令和边界
3. `preheat_only`：只输出小红书 / 朋友圈 / 公众号预热内容
4. `sales_page_only`：只输出销售页骨架
5. `high_risk_boundary_only`：高风险输入时，只输出就医提醒、表达边界和主题重构建议

默认情况下，当用户要的是完整主题课包时，`full_package` 会输出以下 12 个部分：

1. 主题定位
2. 适合人群
3. 不适合人群
4. 用户常见痛点语言
5. 课程目标
6. 课程结构
7. 课程流程
8. 课堂口令方向
9. 表达边界清单
10. 预热内容方向
11. 销售页结构
12. 风险提示

## 示例

示例输入见 [examples.json](./examples.json)，其中包含：

- 久坐后腰背紧的 60 分钟线下课
- 肩颈酸主题的小红书预热内容
- 外伤后膝盖肿胀的高风险边界测试
- 40+ 女性呼吸浅与睡眠差线上课
- 产后骨盆底觉察主题的销售页骨架
- 久站脚累与下肢沉重的朋友圈预热

## 示例覆盖

当前示例已经覆盖三类常见输出：

- 正常课程骨架：如线下课、线上课、直播课
- 内容预热输出：如小红书、朋友圈
- 销售转化场景：如课包销售页
- 高风险边界分流：如外伤、肿胀、活动受限等输入
- 更广的主题覆盖：如呼吸、睡眠、肩颈、腰背、髋部、足部、产后与久站疲劳

如果你要继续扩展，建议优先增加：

- 孕产期低风险与高风险的分界案例
- 睡眠 / 呼吸 / 情绪主题案例
- 肩颈、腰背、髋部、足部等不同部位案例
- 更短时长的 30 分钟体验课案例

## 反馈与提问

建议这个仓库启用 GitHub Discussions，并采用下面的分流方式：

- 使用问题、安装问题、调用方式讨论：走 `Discussions`
- Bug、文档错误、改进建议：走 `Issues`

推荐这样做的原因：

- 使用问题通常需要来回交流，不一定是缺陷
- Bug 和改进建议更适合被追踪、标记和关闭
- 能把“提问”与“待修复事项”分开，仓库会更干净

如果仓库已经开启 `Discussions`，可以在仓库首页引导用户：

- “使用问题请发 Discussions”
- “缺陷反馈和改进建议请提交 Issues”
